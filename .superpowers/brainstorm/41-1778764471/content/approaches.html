Proposed Approaches

Three ways to scope and structure this project. Click your preferred direction.

A

Incremental — Pipeline first, dashboard second

Run the improved Python pipeline (Claude Sonnet + HDBSCAN + richer fields) to produce a new analyzed_incidents_v2.jsonl and a new marine_accident_themes_analysis_v2.md. Compare themes old vs new. Then build the dashboard from the improved data. Two clear phases, easier to check quality at each step.

Pros

  • Can validate theme quality before building dashboard
  • Lower risk — catch data issues early
  • Old and new .md reports directly comparable

Cons

  • Longer before you see anything visual
  • Pipeline changes may force dashboard rework
B

Parallel — Build both simultaneously

Build the improved pipeline and the dashboard at the same time. The dashboard is scaffolded using the existing analyzed_incidents.jsonl data, then hot-swapped to the improved data once the pipeline finishes. Fastest time-to-seeing-something, but requires designing the data contracts upfront.

Pros

  • Fastest overall — see visuals early
  • Dashboard and pipeline developed together

Cons

  • More complex to coordinate
  • Dashboard may need rework if new data shape differs
C

Data-contract first — Define outputs, then build both Recommended

Start by defining the exact JSON schema the dashboard needs (a data contract), then build the pipeline to produce it and the dashboard to consume it. A short prep step that prevents rework in both directions. The pipeline outputs lightweight pre-aggregated JSON files; the dashboard never touches raw CSVs.

Pros

  • Clean separation — pipeline and dashboard are decoupled
  • Easiest to maintain or re-run pipeline later
  • Dashboard loads fast (pre-aggregated, not raw data)
  • Both .md reports (old and new) preserved for comparison

Cons

  • Small upfront design cost before writing code
